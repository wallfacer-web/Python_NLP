# RAG最小工作流

## 方法定位
这张方法卡把 RAG 的最小闭环压缩成 5 步，帮助学生先跑通流程，再考虑优化。

## 适用场景
- 基于课程资料做问答
- 基于私有文档做检索增强问答
- 用小型知识库做课程项目原型

## 不适用场景
- 资料本身极少或极乱
- 问题和资料完全不相关
- 需要高精度专业知识验证但没有人工复核环节

## 五步流程

### 第一步：准备文档
先把文档整理成可读取文本：
- txt
- csv 中的文本列
- pdf/docx 提取后的纯文本

如果文档太乱，先回到文件与语料管理工作台处理。

### 第二步：切块
把长文本切成较小但仍保留基本语义的块。

最小例子：

```python
def split_text(text, size=300):
    return [text[i:i+size] for i in range(0, len(text), size)]
```

这只是最简单的演示版，真正项目里通常会做重叠切块。

### 第三步：建立检索表示
把文本块变成可比较的表示，比如向量。

这一层的重点不是让学生马上精通向量数据库，而是先理解：
- 为什么要把文本变成可计算表示
- 为什么后面才能做“找最相关片段”

### 第四步：检索相关片段
用户提问后，先从文档块里找最相关的几段内容，再把它们送给模型。

逻辑上是：
- 先找资料
- 再让模型回答

### 第五步：生成并核查答案
把问题和检索结果一起交给模型生成答案。
生成后至少检查两件事：
- 答案有没有超出材料乱发挥
- 检索回来的片段是不是本来就不对

## 最小伪代码

```python
documents = load_documents()
chunks = split_documents(documents)
index = build_index(chunks)

question = "课程第几周开始讲文本分类？"
retrieved = retrieve(index, question)
answer = generate_answer(question, retrieved)

print(answer)
```

## 常见错误
- 文档没清洗就直接入库
- 切块太大或太小
- 只看最终答案，不看检索结果
- 以为 RAG 自动等于零幻觉
