# RAG工作台

## 工作台定位
前沿层。负责把“检索”与“大模型生成”结合起来，构建一个更可靠的问答流程。

如果前一页的大模型文本处理更像：
- 直接问模型
- 看模型怎么回答

那么这一页解决的是：

如何先给模型找资料，再让它回答。

## 它属于哪个层级？
- 前沿层

- 输出失败案例与误差分析

## 一、什么是 RAG？
`RAG` 的全称是：

`Retrieval-Augmented Generation`

中文通常说：
- 检索增强生成

它的核心思想是：
- 先找资料
- 再回答问题

## 二、它和普通大模型问答有什么不同？
可以用一个最简单的比喻：

### 普通大模型问答
像闭卷考试。

模型只能靠自己训练时记住的内容回答问题。

### RAG 问答
像开卷考试。

系统会先去资料库里找相关内容，再把资料和问题一起交给模型，让模型基于这些资料回答。

所以它和普通问答最大的区别是：
- 普通问答：主要靠模型记忆
- RAG：先检索外部资料，再生成答案

## 三、为什么 RAG 常被看作减少幻觉的重要方法？
因为它在一定程度上强迫模型：
- 先参考外部证据
- 再组织答案

这就比纯靠“记忆式回答”更稳。

它特别适合：
- 企业内部资料
- 私有课程材料
- 最新文档
- 不在模型训练数据里的内容

所以 RAG 有两个很重要的价值：
- 减少幻觉
- 使用私有资料

## 四、RAG 的最基础流程是什么？
一个最小的 RAG 流程通常包含：

1. 文档准备  
2. 文档切块  
3. 向量化  
4. 检索  
5. 把检索结果放进 Prompt  
6. 生成答案  

这一页最重要的不是记住英文术语，而是理解这条流水线。

## 五、怎么通俗理解 chunk、embedding、retrieval？
### 1. Chunk
`chunk` 可以理解成：
- 从长文档里切出来的一张张知识卡片

因为长文档太长，模型一次不一定读得完，所以通常要先切成小块。

### 2. Embedding
`embedding` 可以理解成：
- 把文本翻译成机器更容易比较的数字坐标

这些坐标不是给人看的，而是给系统用来判断“哪段内容和问题更接近”。

### 3. Retrieval
`retrieval` 就是：
- 从很多知识卡片里找出最相关的几张

所以可以把整个 RAG 理解成：
- 先把资料切成卡片
- 给每张卡片做语义坐标
- 提问时找出最相关的卡片
- 再让模型基于这些卡片回答

## 六、为什么 RAG 不等于永远正确？
这是这一页最关键的边界。

RAG 只是“更可靠”，不等于“永远正确”。

它仍然会出三类典型问题：

### 1. 检错
系统找出来的资料本身就不对。

### 2. 漏检
真正有用的资料没被找出来。

### 3. 答偏
即使资料找对了，模型也可能没按资料好好回答。

所以 RAG 不是终点，仍然需要：
- 检查检索结果
- 检查最终答案
- 做错误分析

## 七、一个最小 RAG 思路示例
这里先用伪代码理解流程，不纠结具体平台细节。

```python
documents = load_documents("course_notes.txt")
chunks = split_text(documents, chunk_size=500, chunk_overlap=50)
vectors = embed(chunks)

question = "这门课最核心的学习目标是什么？"
retrieved_chunks = retrieve(question, vectors, top_k=3)

prompt = f"""
请只根据以下资料回答问题：

资料：
{retrieved_chunks}

问题：
{question}
"""

answer = llm(prompt)
print(answer)
```

这个例子最适合课堂讲：
- RAG 并不是一个神秘黑箱
- 它就是“检索 + 生成”

## 八、一个更贴近课堂的 LangChain 思路示例

```python
from langchain.document_loaders import TextLoader
from langchain.text_splitter import CharacterTextSplitter

loader = TextLoader("my_notes.txt")
documents = loader.load()

text_splitter = CharacterTextSplitter(chunk_size=500, chunk_overlap=50)
docs = text_splitter.split_documents(documents)

print("切块完成：", len(docs))
```

这段代码虽然没有把整个 RAG 全跑通，但它能让学生先看到：
- 长文档如何变成很多块

## 九、为什么切块很重要？
切块大小不合适，会直接影响检索质量。

### 切得太小
问题：
- 上下文不完整
- 句子被切碎
- 语义信息不够

### 切得太大
问题：
- 检索不够精细
- 浪费上下文长度
- 把不相关信息一起塞进去了

所以切块是一个非常实际的工程问题。

### 练习4：失败分析
假设系统答错了，可能是哪一层出了问题？
- 文档太脏
- 切块不合理
- 检索没找到
- 模型没按资料回答

## 十一、初学者最常见错误
### 1. 以为 RAG 一上就能完全消灭幻觉
它只能降低风险，不能保证完全正确。

### 2. 文档没清洗就直接入库
如果资料本身很乱，后面的切块和检索都会受影响。

### 3. 切块尺寸乱设
太大、太小都可能带来问题。

### 4. 只看最终答案，不看检索结果
教学里必须强调：
- 先看找到了什么
- 再看回答得对不对

### 5. Prompt 不够严格
如果不明确要求：
- 只根据给定资料回答
- 不知道就说不知道

模型仍然可能自由发挥。
