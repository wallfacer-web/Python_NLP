# 网站发布说明

这份课程库已经按“公开站”和“内部库”分开考虑。
建站时，先保证内容边界正确，再接 Quartz 和 GitHub Pages。

## 当前已经落实的原则

- 首页和 README 已分开
- 公开站入口使用 `index.md`
- 公开页优先使用 wikilink 和相对路径
- 本地运行痕迹和内部工具目录通过 `.gitignore` 排除

## 不应直接公开的内容

- `.obsidian/`
- `.smart-env/`
- `notebooklm-py/`
- 本地技能说明和内部执行文件

## 公开站建议入口

- [[index]]
- [[课程介绍（网页版）]]
- [[角色入口总览（网页版）]]
- [[工作台总览（网页版）]]

## Quartz 发布原则

- Quartz 适合发布 Markdown、双链和知识导航
- 关键入口页要用 Markdown，不依赖 `.canvas`
- 网站首页必须门户化，不直接拿 README 顶上去
- 公开页要优先服务教师、学生和外部访问者

## GitHub Pages 发布建议

在接入 Quartz 前，先确认三件事：

1. 仓库边界已经清楚，内部目录不会被误公开
2. 网站首页、课程介绍、工作台总览已经准备好
3. README 和公开站首页已经分开

## 当前已经接入的工程能力

- Quartz v4 核心引擎已接入仓库
- 公开内容通过 `scripts/export-public-content.ps1` 导出到 `content/`
- GitHub Pages 工作流已写入 `.github/workflows/deploy.yml`

## 当前发布策略

当前采用“课程内容全量公开，内部工具目录排除”的策略：

- 默认导出根目录中的课程页面
- 默认导出课程内容目录
- 排除 `.obsidian`、`.smart-env`、`notebooklm-py`、`quartz`、`scripts`、`node_modules`、`public`、`content` 等内部或构建目录

也就是说，后续新增课程内容时，原则上不需要反复修改导出脚本。

## 本地使用

```bash
npm install
npm run site:dev
```

构建静态输出：

```bash
npm run site:build
```

## GitHub Pages 启用方式

在 GitHub 仓库设置中：

1. 打开 `Settings -> Pages`
2. 在 `Build and deployment` 中选择 `GitHub Actions`
3. 推送到 `main` 分支后，工作流会自动构建并发布
